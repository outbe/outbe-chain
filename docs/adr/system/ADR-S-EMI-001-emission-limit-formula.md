# ADR-S-EMI-001: EmissionLimit owns the deterministic daily cap and allocation table

- **Status:** Proposed; current implementation profiled
- **Date:** 2026-07-17
- **Decision owners:** Protocol economics maintainers
- **Scope:** `crates/system/emissionlimit`
- **Depends on:** ADR-B-CNS-003, ADR-B-EVM-004, ADR-S-CYC-001
- **Related:** ADR-C-MET-001, ADR-C-AGR-001, ADR-C-PRM-003, ADR-C-GEM-002, ADR-S-RWD-001 and a future daily-emission PFS
- **Supersedes:** EmissionLimit portions of former broad pre-space Cycle/daily-orchestration document (previously numbered 029)

## Context

EmissionLimit defines how much new value a UTC day may allocate and how that cap is
partitioned among named sinks. It is a deterministic economics library with no
independent timer: Cycle invokes the daily command, while Rewards, AgentReward and
Metadosis own delivery state. Keeping formula/allocation separate from scheduling
and recipient ledgers makes parameter changes and conservation auditable.

## Decision

### Closed-form daily cap

For `day_number` since the chain's canonical genesis UTC day:

```text
cap(day) = max(FLOOR_DAY_EMISSION,
               INITIAL_DAY_EMISSION * exp_fp(-K_SOFT * day))
```

Production uses unsigned `U256` fixed-point scale `10^18`, a 32-term alternating
Taylor expansion and integer floor division. At and beyond day 2920 it returns the
fixed floor directly. No storage, wall time or floating point participates. The
caller derives `day_number` from chain-spec genesis and typed UTC dates.

### Static sink allocation

The fork-governed sink table is:

| Sink | Fixed share |
|---|---:|
| Validator | 4% |
| WAA | 4% |
| SRA | 4% |
| CCA | 4% |
| Merchant | 4% |
| Metadosis | terminal remainder |

Every fixed share rounds down. The single final terminal sink receives the entire
unallocated percentage and division dust. Therefore allocation amounts always sum
exactly to the input cap.

The table validator requires exactly one terminal entry in final position and fixed
percentages no greater than 100. Sink ids, order, percentages and dust ownership are
protocol parameters, not runtime plugins.

### Dispatch boundary

The production daily orchestrator owns calls to sinks and completion/replay state.
EmissionLimit may provide pure allocation and a narrow terminal-Metadosis helper,
but it does not own sink balances, retries or `daily_settled`. Any generic
per-sink fallback dispatcher is non-normative unless the production command adopts
and tests that failure model explicitly.

## Invariants

- Cap equals the pinned formula for every `u32` day and never increases.
- Day zero equals initial emission; threshold and later days equal the floor.
- Each fixed allocation is `floor(cap * pct / 100)`.
- Sum of all allocations equals cap exactly.
- There is exactly one terminal sink, last in order, receiving all dust.
- Formula/allocation are pure and byte-deterministic across platforms/builds.
- Caller applies one cap for one canonical UTC day at most once; this replay
  invariant belongs to the daily orchestration/Rewards completion state.

## Arithmetic, compatibility and activation

Constants, scale, Taylor term count/order, unsigned positive/negative accumulation,
rounding, floor threshold, sink ids/order and percentages are consensus economics.
Any change requires an activation height, new complete reference vectors and an
economic supply-impact analysis. Tests generated by the implementation itself are
regression pins, not an independent mathematical oracle.

Input bounds currently make stated products fit `U256`, but every intermediate must
be proven over all `day_number` and cap values. Saturation is not acceptable if it
can change the curve silently.

## Atomicity, failure and evidence

Pure functions either return a valid result or a typed error for invalid custom
tables. Delivery occurs inside the caller's daily transaction/checkpoint. A sink
failure must follow the one accepted orchestration policy and cannot cause cap value
to disappear or mint twice.

Inspected formula, allocation validator/table, terminal dispatch helper and unit
tests for pinned samples, monotonicity, floor, dust and invalid tables. A complete
independent reference model, all-day sweep and production daily e2e are missing.

## Consequences

EmissionLimit remains a small deterministic policy module. It can receive an architecture-conformance
verdict for arithmetic/interface closure without absorbing Rewards, AgentReward,
Metadosis or Cycle state.

## Rejected alternatives

- **Per-block emission:** current economics and settlement authority are daily.
- **Floating-point exponential:** platform/compiler behavior is unsuitable for
  consensus.
- **Drop division dust:** supply conservation fails.
- **Dynamic sink plugins:** runtime registration could change economics without a fork.

## Open questions and technical debt

1. Taylor positive/negative sums and term multiplication use unchecked arithmetic;
   formally prove bounds for all days or use checked operations.
2. `saturating_sub` in the exponential can convert an invalid approximation into
   zero/floor. Make negative/overflow approximation an explicit invariant failure.
3. Golden values are described as generated by the implementation itself. Add an
   independent high-precision reference and committed cross-language vectors.
4. Exhaustively test monotonicity and floor bounds across all days `0..=2920`, not
   only the first year and threshold samples.
5. Validate fixed-share multiplication with checked arithmetic for arbitrary test
   tables/input, even if production cap is bounded.
6. `dispatch_allocations` implements best-effort per-sink fallback while the
   production Cycle handler is all-or-nothing. Remove/relocate it or adopt one
   explicitly tested failure contract.
7. Terminal dispatch date anchoring must distinguish finalized-block UTC date from
   executing-block date and WorldwideDay; add boundary tests across midnight.
8. CCA and Merchant sink ownership/withdrawal FSM remain unspecified; a percentage
   constant alone does not prove usable or recoverable funds.
9. Define governance and activation process for coefficients, floor and sink table.
10. Add a daily-emission PFS proving cap allocation, every sink result/residue,
    replay, finality, failure injection and restart.
11. Prove cumulative issuance over the full intended horizon and reconcile it with
    native/Promis/Gratis/Gem supply models.
12. Clarify whether a zero cap is a successful processed day; current orchestration
    historically advanced trigger without a daily completion marker.
