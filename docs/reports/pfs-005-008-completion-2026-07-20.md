Two Goal phrases require an explicit mapping rather than a fictitious state:

- Update approval and scheduling are one atomic end-block transition. There is
  no reachable persisted state “approved but not yet scheduled” at which a
  process can be restarted. The suite instead proves atomic exactly-once
  approval/scheduling, restarts the full committee immediately after that
  committed boundary, and verifies both records on every validator. A test that
  manufactured an intermediate state would not exercise the protocol.
- The follower recovery scenario first creates real finalized lag by removing
  its only upstream, then changes upstream and restarts the follower from its
  durable datadir as catch-up begins. This is the reproducible restart-during-
  recovery form of the requested catch-up scenario; final hash and state-root
  parity are required before it passes.

## Per-scenario operational log audit

The Cucumber `after` hook calls `Localnet::audit_unexpected_logs` after every
scenario and before teardown. It recursively reads `node.log`, `enclave.log`
and `reth.log`. A scenario fails on any unexpected:

- fatal record;
- panic;
- DKG share reveal/disclosure;
- VRF alarm;
- SGX `EAGAIN` or “resource temporarily unavailable” record;
- projection fatal.

The unsupported-version and intentionally revealed frozen-target scenarios
allow only their exact expected record and require it on every validator;
unrelated alarms still fail. Since all 30 scenarios passed in each canonical
run, all 60 per-scenario audits passed. The harness currently does not print
explicit zero counters, and successful cleanup removed the raw run directories.
Thus the hook result is authoritative pass/fail evidence, while independently
recountable per-scenario log files are not retained. Printing an audit summary
before cleanup would strengthen future release evidence.
